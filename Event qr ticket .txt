```python
ROLE_CODE_MAP = {
    "ADM": "admin",
    "OWN": "apartment",
    "VND": "vendor",
    "SEC": "security",
    "VST": "visitor",
    "EVT": "event_ticket",
    "PTL": "patrol_location",
}

def parse_qr_payload(qr_data: str) -> dict:
    parts = qr_data.strip().split(" - ")
    if len(parts) == 3:
        society_id, role_code, entity_id = parts
        return {
            "society_id": int(society_id),
            "role_code": role_code.strip(),
            "role": ROLE_CODE_MAP.get(role_code.strip()),
            "entity_id": int(entity_id.strip()),
        }
    return {"error": "Invalid format. Expected: society_id - ROLE entity_id"}

def generate_qr(society_id: int, role_code: str, entity_id: int):
    payload = f"{society_id} - {role_code} {entity_id}"
    # same qrcode generation...
    return img_base64, payload

def validate_qr_code(qr_data: str, society_id=None) -> dict:
    parsed = parse_qr_payload(qr_data)
    if "error" in parsed:
        return {"status": "FAIL", "reason": parsed["error"], "gate_action": "deny"}

    if society_id and parsed["society_id"] != society_id:
        return {"status": "FAIL", "reason": "QR not valid for this society", "gate_action": "deny"}

    role = parsed["role"]
    entity_id = parsed["entity_id"]

    # Dispatch to role-specific validators
    if role == "event_ticket":
        return validate_event_ticket_qr(entity_id, society_id)
    elif role == "visitor":
        return validate_visitor_qr(entity_id, society_id)
    elif role == "patrol_location":
        return validate_patrol_qr(entity_id, society_id)
    else:
        return validate_standard_user_qr(entity_id, role, society_id)
```

```python
    """Check event_tickets row + events event_date vs today + used/unused."""
    row = db._execute("""
        SELECT et.*, e.title, e.event_date, e.venue,
               s.name as society_name
          FROM event_tickets et
          JOIN events e ON e.id = et.event_id
          JOIN societies s ON s.id = et.society_id
         WHERE et.id = %s AND et.society_id = %s
           AND et.status = 'active'
    """, (event_ticket_id, society_id), fetch_one=True)

        return {"status": "FAIL", "reason": "Ticket not found or already used", "gate_action": "deny"}
    if row["event_date"] != datetime.now().date():
        return {"status": "FAIL", "reason": f"Ticket valid only on {row['event_date']}", "gate_action": "deny"}
```

## Key Decision Points

### QR Payload Human-Readability
Your proposed format `1-EVT-42` is more readable than `42|EVT|1`. The `-` separator is a good choice because:
- It matches your role abbreviations exactly
- It survives printing/stains better (shorter payload)

### Visitor Walk-in vs Presumptive
I recommend two status flows:
```
PRESUMPTIVE (owner-issued): pending -> approved -> entered -> exited
WALK-IN (security-issued):    pending -> owner_confirm -> entered -> exited
```
Both produce a QR, but walk-ins have a `security_captured_photo` and require real-time owner confirmation.

### Event Ticket QR Scope
Consider limiting ticket QR validity:
- Date only (entire event day)
- Optional 3-hour window from `event_time`
- Mark `status='used'` after first scan so QR cannot be duplicated

### Patrol QR
Bulletproof design:
- `patrol_locations.qr_payload` is fixed (e.g. `1-PTL-5`)
- Security scans it at every shift change and mid-point
- Backend alerts when `time since last scan > scan_interval`
- Patrol performance = % of required scans completed on time

Here are my suggestions on the School Bus / Taxi / Visitor KPI-alert system, design-only:

## 1. Treat this as one generic "Subscribable Alert" system, not three features

School Bus, Taxi, and Visitor confirmation all follow the same shape: **an entity has subscribers → a trigger event fires → subscribers get pushed a notification → a KPI card changes color → a second action (press/click) either escalates or resolves it.** If you build three separate tables and three separate callback trees, you'll be maintaining triplicate logic forever. I'd model:

- `alert_channels` — one row per School Bus or Taxi (`channel_type` = 'school_bus'|'taxi', `name`, `number`, `created_by_apartment_id`, `society_id`, `active`)
- `alert_subscriptions` — `channel_id`, `apartment_id` (who subscribed)
- `alert_events` — one row per press/trigger (`channel_id` or `visitor_id`, `state`, `triggered_by_user_id`, `triggered_at`, `resolved_at`, `resolved_by_user_id`)

Visitor confirmation reuses the same `alert_events` pattern but points at `visitor_id` instead of `channel_id` — same state machine, different foreign key.

## 2. State machine per alert type

**School Bus:** `idle → arrived (green)`. This is announcement-only, no second escalation needed. Auto-expire back to idle after N minutes (configurable, e.g. 10–15) so the card doesn't stay green all day — otherwise tomorrow's bus arrival won't be visually distinguishable from yesterday's. Auto-expire is a background job or a TTL check on render, not a security action.

**Taxi:** `idle → arrived (orange) → [second press] → calling`. The second press shouldn't just change color — it should initiate the actual call. Since you're on a phone-based platform, "calling the owner" from security's device most naturally means launching a `tel:` link with the owner's registered mobile number, not a VoIP bridge (unless you already have Twilio/similar wired in, which would be a bigger lift). Worth confirming which you intend, since it changes the KPI card's second-press behavior meaningfully — a client-side deep link vs. a backend telephony call.

**Visitor (presumptive list):** `pending → security_flagged (orange) → [second press or owner click] → confirmed (green) / denied`. This one is genuinely two-actor: security's second press should represent "I attempted verbal confirmation" (a log entry, maybe with a note field), not the actual approval. The approval authority stays with the owner clicking their push notification. This distinction matters for liability — you don't want security's UI to have a path that admits a visitor without an owner ever having acted.

## 3. Who can flip the card back, and what happens to no-response

Decide upfront what happens if:
- Owner never opens the push (phone off, app backgrounded). Does the visitor entry sit "orange" indefinitely, or timeout to `denied` after a threshold? I'd suggest a timeout to `denied`/`pending_recheck` after e.g. 5–10 minutes, with security able to re-trigger.
- Multiple apartment owners are linked to one flat (spouse + owner both have login) — either should be able to confirm; first click wins, second click should show "already confirmed by X" rather than erroring.
- Security presses the KPI a third time after "calling" state — decide if that's a no-op, a re-call, or an escalation to admin. I'd make it a no-op with a toast ("already escalated") to avoid duplicate calls.

## 4. KPI card rendering — piggyback on your existing dynamic KPI system

You already have `_kpi_row_dynamic()` and `society_id`-threaded KPI infrastructure from the customization work. I'd extend that same mechanism rather than building bespoke cards:
- Add `kpi_active_school_bus_alerts`, `kpi_active_taxi_alerts` as new KPI types that query `alert_channels` joined to the latest `alert_events` row per channel, colored by state.
- This gets you layout customization, portal filtering, and permission scoping for free instead of reinventing it for this feature.

## 5. Push notification targeting

Your `push_service.py` already has `get_notification_targets()`. Extend its targeting logic to resolve "all apartments subscribed to channel X" as a first-class target type, alongside whatever role/apartment targeting it already supports. Keep the notification **creation** (which you fixed to always run unconditionally before push delivery) as the audit trail — even if push fails, the in-app notification/KPI state should still update, since color-state and push are two separate concerns that shouldn't be coupled to the same success/failure path.

## 6. Race conditions on the second press

Multiple security guards could theoretically be logged in on shared devices at the gate. Use a conditional update (`WHERE state = 'arrived'` in the UPDATE clause, checking rowcount) rather than read-then-write, so two simultaneous "call" presses don't both fire duplicate calls or double-log events.

## 7. Reuse your existing QR flow for visitor mobile scan-in

You mentioned security can either press the KPI or scan the visitor's QR from their phone. Since `qr_service.py` already dispatches on the `V` role code to `validate_visitor_qr`, the QR scan path and the KPI-press path should converge on the **same** state-transition function (something like `flag_visitor_for_confirmation(visitor_id)`), rather than duplicating the orange-transition logic in two places. The QR scan just becomes an alternate *entry point* into the same state machine as the manual KPI press.

## 8. Open decision I'd nail down before building

Do you want Taxi/School Bus channels to be **per-apartment recurring** (owner sets up "Blue Bird Bus #12" once, it persists across days) or **per-day, like the presumptive visitor list** (recreated each morning)? Recurring channels need an `active` toggle and a way to prevent stale subscriptions accumulating; per-day channels need bulk cleanup/archival. Your description ("subscribed by self and other owners") sounds recurring, but worth confirming since it changes the seed/cleanup story.
